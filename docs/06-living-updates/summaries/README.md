# Summaries

Simplified one-page explanations of updates for teammates who don't need the full technical detail.

## When to Write a Summary

Write a summary when:
- An update affects the whole team, not just one role
- The source material (changelog, announcement) is technical or lengthy
- You want a quick-reference version alongside the full impact note

## Format

Keep summaries short. Three sections maximum:

1. **What changed** — one sentence
2. **What it means for us** — two to three sentences
3. **What to do** — bulleted action items, if any

## Naming

```
YYYY-MM-<vendor>-<short-slug>.md
```

Example: `2026-01-github-pr-files-changed.md`

## Relationship to Other Docs

```
incoming/     → raw, unreviewed notes (auto-generated or manual)
summaries/    → short plain-language explanations (human-written)
impacts/      → approved guidance changes (reviewed and merged)
archive/      → reviewed, no action needed
```

A summary is optional. An impact note is required for any update that changes team guidance.
