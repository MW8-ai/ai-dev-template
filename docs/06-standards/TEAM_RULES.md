# Team Rules Expanded

## Team Principle

We optimize for safe speed.

That means:
- Move quickly.
- Keep changes reviewable.
- Automate checks.
- Protect main.
- Leave a clear trail.

## Rules

### 1. No Direct Commits to Main

WHY:
Main should always represent deployable, trusted code.

### 2. Every Change Uses a Branch

WHY:
Branches isolate work and reduce accidental breakage.

### 3. Every Change Uses a Pull Request

WHY:
Pull Requests create review, discussion, CI checks, and history.

### 4. PRs Should Be Small

WHY:
Small PRs are easier to review and safer to merge.

### 5. PRs Must Explain Why

WHY:
Future maintainers need context, not just code.

### 6. CI Must Pass Before Merge

WHY:
Automation catches repeatable problems better than humans.

### 7. Security Issues Are Not Public Issues

WHY:
Publicly posting secrets or vulnerabilities increases risk.

### 8. Code Owners Review Sensitive Areas

WHY:
Some files affect deployment, security, infrastructure, or production behavior.

### 9. Delete Branches After Merge

WHY:
Stale branches create confusion and accidental reuse.

### 10. Respect the Release Process

WHY:
Releases are risk events, not just Git tags.

## Working Agreement

- Ask early.
- Review kindly.
- Explain clearly.
- Automate repeatable checks.
- Document decisions that affect others.

## Golden Rule

> Good teams do not rely on memory. They build systems that guide behavior.
